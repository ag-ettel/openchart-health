# v1.1 Backlog

Captured at v1 launch (2026-05-13). v1 is live at openchart.health on
Cloudflare Pages: home, methodology, compare, filter-explore render
server-side with full SEO; per-provider pages are client-rendered from
Cloudflare R2 via an `_redirects` SPA fallback.

Ordered by impact-to-effort. Each item notes why it was deferred and the
concrete approach.

---

## P1 — Per-provider SEO

**Why deferred:** 22K provider HTML files would exceed Cloudflare Pages'
20K file limit, and the per-provider JSON data is gitignored (it lives in
R2). v1 serves one shell HTML for every `/hospital/*` and `/nursing-home/*`
URL; search engines see no per-provider content until JS executes.

**Impact:** This is the single biggest gap. Provider profile pages are the
long-tail SEO surface — "{hospital name} quality data", "{nursing home}
inspection record" searches. Without indexable per-provider HTML, organic
discovery is limited to the home/compare/explore pages.

**Approach (hybrid pre-render):**
1. At Cloudflare build time, pull `search_index.json` (already ships) and
   select the top N providers by a deterministic traffic proxy — largest
   hospitals by discharge volume, all SFF/abuse-flagged nursing homes,
   and any provider that appears in a curated high-interest list.
   Target N ≈ 3,000–5,000 (well under the 20K limit with room for the
   ~250 other static files).
2. For those N, fetch their JSON from R2 during the build and emit real
   pre-rendered HTML with full metadata + JSON-LD (the original
   server-component path, restored behind a `TOP_PROVIDERS` allowlist).
3. The long tail keeps the SPA-fallback client render.
4. `generateStaticParams` returns the allowlist slugs + the `_unavailable`
   shell; `_redirects` still catches everything not pre-rendered.
5. Sitemap already enumerates all providers from `search_index.json` —
   no change needed there; submit it once and Google will crawl the
   pre-rendered set first.

**Alternative if file count is still tight:** move to Cloudflare Workers
(not Pages) with Static Assets + a Worker that server-renders provider
pages on demand from R2. No file-count limit, true SSR, but a larger
migration. Evaluate only if the hybrid pre-render proves insufficient.

**Effort:** Medium. Mostly restoring the server-component render path
behind an allowlist + a build-time R2 fetch step + selecting the N list.

## P2 — Conversion / funnel analytics

**Why deferred:** Free Cloudflare Web Analytics (chosen for v1, no spend)
is pageviews + web vitals only — no custom events. The 5 typed event
helpers (`compareStarted`, `compareNearbyOpened`,
`compareNearbyResultClicked`, `measureFilterExploreSelected`,
`measureFilterExploreSorted`) are wired into the UI but inert.

**Impact:** Can't answer "do people who land on a provider page go on to
compare?" or "which measures drive filter-explore usage?" — the questions
that inform what to build next.

**Approach:** Cloudflare Workers Analytics Engine (free tier: generous
write volume). Add a tiny Worker endpoint that accepts the event payloads
the existing `lib/analytics.ts` helpers already construct, writes to
Analytics Engine, and query via the GraphQL Analytics API or a small
dashboard. The event call sites already exist — only the sink changes.
The dormant Plausible scaffolding can stay as-is (it's a no-op without
the env var) or be removed once Analytics Engine is proven.

**Effort:** Small–Medium. One Worker, swap the analytics transport in
`lib/analytics.ts`, leave call sites untouched.

## P3 — Accessibility recommendations (audit R-items)

From `docs/audit_a11y_perf.md`. All 10 critical issues (C1–C10) were
fixed pre-launch. These are the non-blocking recommendations:

- **R1 — `next/font/google` migration.** Inter is loaded via a
  render-blocking `<link>` in `layout.tsx`. Self-hosting via
  `next/font/google` cuts CLS and a render-blocking request. Low risk,
  works with static export.
- **R3 — Server-component conversion.** ~8–10 of 35 `"use client"`
  components are pure-render (no state/effects) and could ship 0 JS.
  Note: the v1.1 per-provider SEO work will re-server-componentize the
  profile pages anyway — sequence R3 after P1 to avoid rework.
- **R4 — Dynamic-import Recharts.** `TrendChart` / `CompareTrendChart`
  pull ~90–120KB gzipped eagerly. `next/dynamic` so the bundle loads
  only when a trend `<details>` expands.
- **R5 — Shard `distributions.json`.** Currently a single 2MB file
  fetched on first histogram render. Shard by `measure_id` so a page
  fetches only the 1–5 distributions it needs. Biggest single perf win
  for profile pages.
- **R7 — HCAHPS stacked-bar color-only encoding.** Add a hatch/pattern
  or always-visible micro-legend so the blue/light-blue/gray response
  tiers aren't distinguishable by hue alone (deuteranopia).
- **R9 — `title`-attr tooltips → Radix Tooltip.** ~12 files use
  `title=""` for supplementary info that isn't keyboard-reachable.
  Swap the load-bearing ones to the already-approved Radix Tooltip.
- **R10 — MeasurePicker grouping semantics.** 207 measures in a flat
  list; wrap each `measure_group` in `<fieldset>`/`<legend>` or proper
  heading landmarks for AT navigation.
- **R12 — Tail-risk badge text reinforcement.** Manual visual QA pass
  to confirm no J/K/L citation row leans on orange alone — code review
  said it passes; verify in the live UI.

**Effort:** R1/R4/R5 are small and high-value (perf). R3 sequence after
P1. R7/R9/R10/R12 are small polish.

## P4 — Per-provider OG images

**Why deferred:** Generic `og-default.png` works for launch. Per-provider
social cards (provider name + 1–2 headline stats) materially lift
share-through but aren't load-bearing.

**Approach:** Satori-based OG generation at build time for the same
top-N allowlist from P1 (reuse that selection). One image per
pre-rendered provider; long tail falls back to `og-default.png`.
Sequence after P1 since it shares the allowlist machinery.

**Effort:** Small once P1's allowlist exists.

## P5 — R2 custom domain (cosmetic)

Replace `pub-b1c04c7dc63e4dd7bad1e6516edb0abb.r2.dev` with
`data.openchart.health`. R2 bucket → Settings → Custom Domains → connect
`data.openchart.health` (Cloudflare auto-creates the CNAME) → update
`NEXT_PUBLIC_CDN_BASE` env → add the new origin to the bucket CORS
policy → redeploy. Purely cosmetic; the r2.dev URL works indefinitely.

**Effort:** Trivial (~10 min, mostly DNS propagation wait).

## P6 — Pipeline test coverage gaps

Phase-1 target was 100% on `pipeline/normalize/`,
`pipeline/validate/`, `pipeline/transform/`. Current state (from the
last `pytest` run): normalize 70–95%, validate 100%, transform 100%,
store ~9%, orchestrate 0%. The store/orchestrate gaps are by design
(integration-tested, not unit-tested), but the normalize gap should
close to satisfy the gate. Specific low-coverage modules noted in the
coverage report: `nh_provider_info.py` (~36%), several NH normalizers
70–80%.

**Effort:** Medium. Mechanical — add parametrized cases for the
documented suppression encodings still uncovered.

## Operational follow-ups (not feature work)

- **Pipeline refresh scheduler is built but not installed.** Cron /
  Task Scheduler entries are documented in `docs/refresh_schedule.md`
  but not registered on any host. Install when the first new CMS
  snapshot lands so data doesn't go stale.
- **Automated archive download** is a manual step (see
  `docs/refresh_schedule.md`). Reconcile DKAN distribution structure
  with the per-archive ZIP layout if/when worth automating.
- **Sentry source maps.** Errors are captured but stack traces are
  minified until `SENTRY_ORG` / `SENTRY_PROJECT` / `SENTRY_AUTH_TOKEN`
  are set in Cloudflare Pages env. Add when debugging a real prod issue
  becomes painful.
- **Local Windows build.** `next build` fails on Windows + Node 24
  (readlink edge case on Next internals). Linux CI is unaffected. For
  local verification use WSL or Node 22. Not worth fixing unless local
  full-builds become routine.
